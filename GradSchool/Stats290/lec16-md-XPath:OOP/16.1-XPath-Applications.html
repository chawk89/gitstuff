16.1. XML/XPath applications

Recall XPath expressions are ways to navigate to elements in XML documents. Very much like directory paths in Unix.

The XML document is treated as a tree of nodes with topmost element as root and different kinds of nodes: element, attribute, text, namespace, processing-instruction, comment, and document nodes.

The usual terms such as parent, child, sibling, ancestor, descendants are used to describe nodes.

There is a quick primer here.

A rich set of functions is available to get at nodes including math, string manipulation etc. For details, see this.

Typical XPath use involves a location specification that is then used in an expression. Location specification syntax: axis::node[predicate]

  • node is a node tag, * is a wildcard, text() refers to a text node which have no names
  • axis specifies the direction to look for a node such as: child, ancestor, parent, descendant, etc.
  • predicate is a condition used to select node sets. This could be a condition that specifies a value for a particular attribute, for example.

An XPath expression returns either a node-set, a string, a Boolean, or a number.

We also said that in XML, you can define your own tags. This means that tag name collisions are inevitable. How to handle this? Enter Namespaces.

Namespaces help you to refer to specific contexts of the tags. Namespaces are implemented by attaching a prefix to each element and attribute. Each prefix is mapped to a URI (Uniform Resource Identifier) by an xmlns:prefix attribute. Avoids ambiguity.

The namespace is defined by the xmlns attribute in the start tag of an element. When a namespace is defined for an element, all child elements with the same prefix are associated with the same namespace. For this reason, it is often convenient to define the namespace in the root element of the document.

Not used by parser to look up information, although generally, one expects there to be some pointer to namespace info at the URI.

16.1.1 KEGG Pathways in Bioinformatics

In biology, there is a tremendous amount of pathway information coded in XML.

For example, consider the KEGG pathway for Colorectal Cancer.

There is help for reading this graph; the components of the pathways in KEGG are defined at here.

  • Rectangular box represents a gene product (a protein/RNA molecule)
  • Small circle represents a compound
  • Large oval represents a link to another pathway map
  • Relationships for molecular interactions: +p,-p, +g, +m for phosphorlyation, dephosphorlyation, glycosylation and methylation
  • Rectangular boxes are complexes

16.1.2. Detecting relationships

How can we determine which gene phorphorylates another? For that we need to understand the structure of the XML. So you load it in your browser.

So this XML file has nodes named entry graphics, relation etc.

So perhaps, relation is what we are after.

Let’s work with the KRAS oncogene and its related neighbors.

We need to know more about the KEGG Markup Language.

Obviously, each entry is uniquely identified by its identifier. So for the relation, we need all entries where entry1 has the id of the KRAS gene, which we have to figure out.

The KRAS gene name in the KEGG scheme is well-established to be hsa:3845.

First we find the KRAS gene id. We have to create the path expressions programmatically. Instead of using paste or paste0 it is convenient to use sprintf.

pathExp  <- sprintf("/pathway/relation[@entry1='%s']", "hsa:3845")
cat("Path Expression is", pathExp, "\n")
## Path Expression is /pathway/relation[@entry1='hsa:3845']
ccXML %>%
    xml_find_all(xpath = "/pathway/entry[@name='hsa:3845']") %>%
    xml_attr(attr = "id") ->
    krasGeneId
cat("KRAS gene id in KEGG is", krasGeneId, "\n")
## KRAS gene id in KEGG is 21

Now for relations containing our gene. We first find all the neighbors of the KRAS gene.

pathExp <- sprintf("/pathway/relation[@entry1='%s']", krasGeneId)
cat("path expression is for entry 1", pathExp, "\n")
## path expression is for entry 1 /pathway/relation[@entry1='21']
ccXML %>%
    xml_find_all(xpath = pathExp) %>%
    xml_attr(attr = "entry2") ->
    krasNeighbors

This gets us all neighbours of KRAS who are listed in entry2, but might not be sufficient because there may be genes for which the krasGeneId might be the second entry! So we account for that.

pathExp <- sprintf("/pathway/relation[@entry2='%s']", krasGeneId)
cat("path expression is for entry 2", pathExp, "\n")
## path expression is for entry 2 /pathway/relation[@entry2='21']
ccXML %>%
    xml_find_all(xpath = pathExp) %>%
    xml_attr(attr = "entry1") %>%
    c(krasNeighbors) %>%  ## Combine with previous neigbours
    unique ->  ## Select unique values
    krasNeighbors

Let’s now look at the relationships.

pathExp <- sprintf("/pathway/relation[@entry1='%s' and @entry2='%s']/subtype",
                   krasGeneId, krasNeighbors)
pathExp
## [1] "/pathway/relation[@entry1='21' and @entry2='11']/subtype"
## [2] "/pathway/relation[@entry1='21' and @entry2='22']/subtype"
## [3] "/pathway/relation[@entry1='21' and @entry2='26']/subtype"

If we try the first path expression for example, we get:

xml_find_all(ccXML, pathExp[1])
## {xml_nodeset (1)}
## [1] <subtype name="activation" value="--&gt;"/>

But we’d like to do it all at once rather iterate for reasons of efficiency. We create a compound expression:

cpathExp <- paste(pathExp, collapse = " | ")
cpathExp
## [1] "/pathway/relation[@entry1='21' and @entry2='11']/subtype | /pathway/relation[@entry1='21' and @entry2='22']/subtype | /pathway/relation[@entry1='21' and @entry2='26']/subtype"

This is now easy to apply to get the types of relationships.

xml_find_all(ccXML, cpathExp)
## {xml_nodeset (3)}
## [1] <subtype name="activation" value="--&gt;"/>
## [2] <subtype name="activation" value="--&gt;"/>
## [3] <subtype name="activation" value="--&gt;"/>

So picking off the actual types of relationships is now trivial.

pathExp <- sprintf("/pathway/relation[@entry1='%s' and @entry2='%s']/subtype",
                   krasGeneId, krasNeighbors)
cpathExp <- paste(pathExp, collapse = " | ")
rels <- xml_find_all(ccXML, cpathExp) %>%
    xml_attr(attr = "name")
print(rels)
## [1] "activation" "activation" "activation"

To detect all phosphorylation entries, let us examine something further. This requires looking at all nodes where the subtype has a name phosphorylation.

pathExp <- "/pathway/relation/subtype[@name='phosphorylation']"
relations <- xml_find_all(ccXML, pathExp)
relations

The key point is that the XML format provides a lot of flexibility for future extensions and so while the computing can be tedious at times, it is worth the effort.

16.1.3. Bioconductor example

I wish to point out that the important project Bioconductor relies more on the recommended package XML and so the approaches used there are different.

For example, here is how one would use that for mining protein-protein interactions.

## Requires bioconductor. Install as:
## source("http://bioconductor.org/biocLite.R") followed by
## biocLite()
library(XML)

Protein-Protein interactions are encoded in yeast.xml. We parse and get the default namespaces.

yeast <- xmlTreeParse("yeast.xml", useInternalNodes=TRUE)
nsY <- xmlNamespaceDefinitions(xmlRoot(yeast))
ns <- structure(sapply(nsY, function(x) x$uri), names = c("ns", "xsi"))

## Get interaction detection methods
idm <- getNodeSet(yeast, "//ns:interactionDetectionMethod//ns:fullName",
                  ns)
sapply(idm, xmlValue)
## [1] "two hybrid"

In this case, all the interaction methods are two hybrid which is molecular biology technique for detecting protein-protein interactions.

We can also get the full name of the organism being studied.

fullname <- getNodeSet(yeast, "//ns:hostOrganism//ns:fullName",
                       ns)
sapply(fullname, xmlValue)
## [1] "Saccharomyces cerevisiae"

Now the interactors:

interactors <- getNodeSet(yeast, "//ns:interactorList//ns:interactor",
                          ns)
length(interactors)
## [1] 503

There are 503 protein-protein interactors.

Finally, the actual interactions.

interactions <- getNodeSet(yeast, "//ns:interactionList/ns:interaction",
                           ns)
length(interactions)
## [1] 524

There are 524 protein-protein interactions.

16.1.4. Article Metadata Mining

The Library of Congress uses the Metadata Object Description Schema for describing bibliographic elements. It can be used for a variety of purposes, particularly library applications.

Several open journals distribute their metadata in __MODS_ format. For example, the Journal of Statistical Software distributes a dump of all articles published.

Now suppose we wanted to mine this information. For example, consider finding all co-authors of author Hornik in the JSS. You can use Xpath expressions to get at this.

BTW, XPath has extensive capabilities that I am only briefly describing here for purposes of illustration.)

doc <- read_xml("jstatsoft.xml")
xml_ns(doc)
## d1 <-> http://www.loc.gov/mods/v3
ns <- xml_ns_rename(xml_ns(doc), d1 = "jss")

Get family name nodes and get the text value of the family names.

xpath1 <- "//jss:namePart[@type='family']"
fnameNodes <- xml_find_all(doc, xpath1, ns = ns)
lastNames <- fnameNodes %>%
    xml_contents %>%
    xml_text
head(unique(sort(lastNames)))
## [1] "Achard"     "Adams"      "Admiraal"   "Ahn"        "Al-Subaihi"
## [6] "Alexander"

Recall that the // is a shortcut for anywhere in the tree. Read as: match the nodes no matter where in the tree they are.

If you wanted to focus on just the first author familyname:

## Pick just the first author family name
xpath2 <- "/jss:modsCollection/jss:mods/jss:name[1]/jss:namePart[@type='family']"
cNodes <- xml_find_all(doc, xpath2, ns = ns)
fLastNames <- cNodes %>%
    xml_contents %>%
    xml_text
tail(unique(sort(fLastNames)))
## [1] "Yuan"    "Zamar"   "Zang"    "Zeileis" "Zhao"    "Zhou"

Same as above using position function

xpath3 <- "/jss:modsCollection/jss:mods/jss:name[position() = 1]/jss:namePart[@type='family']"
cNodes <- xml_find_all(doc, xpath3, ns = ns)
fLastNames <- cNodes %>%
    xml_contents %>%
    xml_text
tail(unique(sort(fLastNames)))
## [1] "Yuan"    "Zamar"   "Zang"    "Zeileis" "Zhao"    "Zhou"

Ok, now we find all co-authors of Hornik in JSS excluding Hornik

coAuthPath <- "//jss:namePart[@type='family' and string(.) = 'Hornik']/ancestor::jss:mods/jss:name/jss:namePart[@type='family' and not(./text()='Hornik')]/text()"

Read as follows: - Find all namePart nodes whose family name is Hornik - For each such node, find the ancestor (i.e. go up one level in the tree) and grab all the family names from the namePart nodes ensuring that Hornik is not in them.

cNodes <- xml_find_all(doc, coAuthPath, ns = ns)
cNodes %>%
    xml_text %>%
    unique
##  [1] "Zeileis"     "Leisch"      "Kleiber"     "Karatzoglou" "Smola"      
##  [6] "Hahsler"     "Grün"        "Meyer"       "Buchta"      "Feinerer"   
## [11] "Hothorn"     "Wiel"        "Leeuw"       "Mair"
